Principal Engineer Roadmap

  The audit says AntriMedis already has a strong MVP backbone: Flutter patient app, Supabase Auth/Postgres/RPC/RLS/Realtime, local
  notifications, queue lifecycle, profile completion, and history. The right strategy for a solo developer is not to re-architect now. Ship the
  patient MVP by hardening the risky edges first: secrets, security, queue correctness, production debugging, and beta-ready reliability.

  Do not prioritize Riverpod/BLoC migration, full admin architecture, predictive queue estimates, offline sync, or multi-branch scale until the
  MVP is actually deployed and observed.

  Phase 1: Immediate

  Goal: make the current MVP safe enough for internal QA and real Supabase usage.

  1. Remove and rotate committed/demo credentials
      - Impact: Very High
      - Effort: Low
      - Risk: Very High if ignored
      - Priority: P0

     Treat anything referenced in .env.example, supabase/README.md, or docs/current_project_snapshot.md as compromised if it points to a real
     project. For a solo developer, this is the highest-value fix because it is small and removes a serious deployment risk.

  2. Stop bundling .env as a Flutter asset
      - Impact: Very High
      - Effort: Low-Medium
      - Risk: High
      - Priority: P0

     The audit says .env is included as a Flutter asset. The anon key is not secret, but this creates a bad path for accidentally shipping non-
     client secrets later. Use a clearly client-only config path or build-time defines.

  3. Separate service-role usage from client app configuration
      - Impact: High
      - Effort: Low
      - Risk: High
      - Priority: P0

     SUPABASE_SERVICE_ROLE_KEY being present in templates, even blank, is risky because it teaches the wrong pattern. The Flutter app should
     only ever know client-safe config.

  4. Replace local PIN storage
      - Impact: High
      - Effort: Medium
      - Risk: High
      - Priority: P0

     SharedPreferences plus unsalted SHA-256 is not adequate for a real app lock. Move PIN material to flutter_secure_storage. Biometric or OS-
     backed auth can follow, but secure storage should happen first.

  5. Supabase security audit
      - Impact: Very High
      - Effort: Medium
      - Risk: Very High
      - Priority: P0

     The database is already doing the right kind of work: RPCs, RLS, triggers, views, and realtime. Before real users, verify RLS policies,
     storage bucket policies, RPC grants, role checks, security_invoker, and realtime scope.

  Phase 2: Before Beta

  Goal: make queueing reliable enough that real patients can use it without the developer manually babysitting every failure.

  1. Add focused tests around core Flutter logic
      - Impact: High
      - Effort: Medium
      - Risk: Medium-High
      - Priority: P1

     Add tests for queue models, friendly error mapping, QueueProvider, AuthProvider, and ProfileProvider. Do not chase broad coverage. Test the
     flows that can break patient access or queue state.

  2. Add queue lifecycle integration tests or repeatable Supabase fixtures
      - Impact: Very High
      - Effort: Medium-High
      - Risk: High
      - Priority: P1

     Cover create, call, serving, missed, recall, skipped, cancelled, and expired. The audit says the SQL layer is robust, but the documented
     rules are not encoded in tests.

  3. Add concurrency testing for create_queue_ticket
      - Impact: High
      - Effort: Medium
      - Risk: High
      - Priority: P1

     Queue creation is the race-sensitive feature. A solo developer does not need a full load-testing platform, but does need a repeatable
     script that proves duplicate or invalid tickets are blocked under pressure.

  4. Improve production error visibility
      - Impact: High
      - Effort: Medium
      - Risk: Medium-High
      - Priority: P1

     Some provider/realtime exceptions are swallowed. That protects UX but hurts diagnosis. Add a minimal app logging/error reporting path
     before beta.

  5. Remove single-clinic hardcoding from the UI
      - Impact: Medium-High
      - Effort: Medium
      - Risk: Medium
      - Priority: P1

     The app currently assumes one primary branch and hardcoded clinic identity/hours. For MVP, it can still be single-clinic, but the displayed
     clinic name and hours should come from backend configuration.

  6. Align client date/time behavior with backend Jakarta-time rules
      - Impact: High
      - Effort: Medium
      - Risk: High
      - Priority: P1

     The audit specifically flags ScheduleAvailability using device-local DateTime.now while the backend uses Jakarta time. That can produce
     wrong queue state for users with different device timezones.

  Phase 3: Production

  Goal: support real-world operation after beta without turning the solo project into a rewrite.

  1. Add production notification architecture
      - Impact: High
      - Effort: High
      - Risk: Medium-High
      - Priority: P2

     Local notifications are not enough when the app is terminated. Add FCM device tokens, Edge Functions or database-triggered delivery, and
     notification delivery status. This is production-level work, not a Phase 1 blocker.

  2. Split QueueProvider incrementally
      - Impact: Medium-High
      - Effort: Medium
      - Risk: Medium
      - Priority: P2

     Do not migrate the whole app. Start by separating schedule state, active ticket tracking, history/timeline state, and notification side
     effects. This lowers test friction and reduces lifecycle bugs.

  3. Introduce mock-friendly repository boundaries where tests need them
      - Impact: Medium
      - Effort: Medium
      - Risk: Medium
      - Priority: P2

     The audit says there is no strong dependency injection boundary. Add seams only where tests need them, especially around repositories/
     providers. Avoid building a large architecture framework.

  4. Introduce typed routing for known painful flows
      - Impact: Medium
      - Effort: Medium-High
      - Risk: Medium
      - Priority: P2

     Manual navigation is acceptable for MVP, but OAuth callback, password recovery, tracking detail, and future role flows will benefit from
     something like go_router. Do this once beta flows stabilize.

  5. Review avatar upload and profile photo privacy
      - Impact: Medium
      - Effort: Low-Medium
      - Risk: Medium
      - Priority: P2

     Add client-side size/dimension validation. Public avatar URLs may be acceptable for MVP, but the privacy decision should be explicit before
     production.

  Phase 4: Scale

  Goal: handle product growth only after the MVP proves usage.

  1. Generate or type Supabase schema contracts
      - Impact: Medium
      - Effort: Medium
      - Risk: Medium
      - Priority: P3

     Manual JSON parsing is fine early, but it gets fragile as views evolve. Add typed contracts once schemas are stable enough.

  2. Clarify the effective database schema
      - Impact: Medium
      - Effort: Medium
      - Risk: Medium
      - Priority: P3

     The audit says migrations are extensive and iterative. Keep the migration history, but create a clear current-schema reference High: P3

     Provider is good enough for MVP. Moving to Riverpod/BLoC may eventually help multi-role, multi-branch, background sync, and offline
     restore, but doing it early would consume limited solo-dev time.

  3. Add Edge Functions or a service layer only where direct Supabase becomes insufficient
      - Impact: Medium
      - Effort: High
      - Risk: Medium
      - Priority: P3

     The direct-from-client Supabase model is efficient for MVP. Add server-side workflow orchestration only when a specific workflow needs it.

  4. Improve queue estimates
      - Impact: Low-Medium
      - Effort: Medium-High
      - Risk: Low
      - Priority: P4

     Current average-service-minute estimates are acceptable for MVP. Historical or predictive estimates are scale work, not launch work.

  Opinionated Solo-Developer Cut

  Build and deploy with the current architecture. Do not pause for a state-management rewrite, routing rewrite, admin-panel rebuild, or
  predictive queue model.

  The real MVP blockers are:

  1. Secret/config safety.
  2. Local PIN storage.
  3. Supabase security audit.
  4. Queue lifecycle and concurrency tests.
  5. Production-visible errors.
  6. Timezone correctness.
  7. Reliable notification delivery before full production.

  Everything else should wait until beta usage proves the need.
